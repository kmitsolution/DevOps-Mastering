# Introduction to Continuous Integration (CI) with GitLab

## What is Continuous Integration (CI)?

**Continuous Integration (CI)** is a software development practice where developers frequently integrate their code changes into a shared GitLab repository. Every time code is pushed, GitLab automatically executes a series of predefined tasks such as building the application, running tests, generating reports, and verifying that the application is working correctly.

The main objective of CI is to detect issues early, improve code quality, and ensure that the application remains in a deployable state throughout development.

GitLab provides a built-in CI system that automatically creates and executes pipelines whenever changes are pushed to the repository.

---

# How GitLab CI Works

When a developer pushes code to a GitLab repository, the following sequence occurs:

1. GitLab detects the new commit.
2. GitLab reads the **`.gitlab-ci.yml`** file from the repository.
3. GitLab creates a pipeline.
4. The pipeline is divided into one or more stages.
5. Each stage contains one or more jobs.
6. GitLab assigns each job to an available GitLab Runner.
7. The Runner executes the commands defined in the `script` section.
8. GitLab displays the logs and job status in the Pipeline page.

---

```text

        ▼

```

# Creating Your First CI Pipeline

Create a file named:

```text
.gitlab-ci.yml
```

Place it in the root directory of your GitLab repository.

---

# Simple CI Pipeline

```yaml
stages:
  - build
  - test
  - report

build-job:
  stage: build
  script:
    - echo "========== BUILD STAGE =========="
    - echo "Downloading source code..."
    - echo "Compiling the application..."
    - echo "Build completed successfully."

test-job:
  stage: test
  script:
    - echo "========== TEST STAGE =========="
    - echo "Running unit tests..."
    - echo "Running integration tests..."
    - echo "All tests passed successfully."

report-job:
  stage: report
  script:
    - echo "========== REPORT STAGE =========="
    - echo "Generating test report..."
    - echo "Collecting build logs..."
    - echo "CI Pipeline completed successfully."
```

---

# Understanding the Pipeline

Let's understand every component of the pipeline.

---

## 1. stages

```yaml
stages:
  - build
  - test
  - report
```

The **stages** section defines the different phases of the pipeline.

In this example, the pipeline consists of three stages:

* Build
* Test
* Report

GitLab executes stages **sequentially**, meaning the next stage starts only after the previous stage completes successfully.

Execution order:

```text
Build
   │
   ▼
Test
   │
   ▼
Report
```

---

## 2. Jobs

```yaml
build-job:
```

```yaml
test-job:
```

```yaml
report-job:
```

A **job** is an individual task executed by the GitLab Runner.

Each job belongs to exactly one stage.

In this example:

| Job        | Stage  |
| ---------- | ------ |
| build-job  | Build  |
| test-job   | Test   |
| report-job | Report |

---

## 3. Stage Assignment

```yaml
stage: build
```

This line tells GitLab that the job belongs to the **Build** stage.

Similarly,

```yaml
stage: test
```

assigns the job to the Test stage.

```yaml
stage: report
```

assigns the job to the Report stage.

---

## 4. Script

```yaml
script:
```

The **script** section contains the Linux commands that the GitLab Runner executes.

Each command begins with a hyphen (`-`).

Example:

```yaml
script:
  - command1
  - command2
  - command3
```

GitLab Runner executes these commands from top to bottom.

# Build Stage

```yaml
build-job:
  stage: build

  script:
    - echo "========== BUILD STAGE =========="
    - echo "Downloading source code..."
    - echo "Compiling the application..."
```

This stage simulates the build process.

Although no actual compilation occurs, the `echo` statements represent common build activities such as downloading source code and compiling the application.

Expected output:

```text

Downloading source code...
Compiling the application...
Build completed successfully.
```

# Key Concepts

| Component            | Description                                                      |
| -------------------- | ---------------------------------------------------------------- |
| **Pipeline**         | The complete CI workflow executed by GitLab.                     |
| **Stage**            | A logical phase of the pipeline, such as Build, Test, or Report. |
| **Job**              | An individual unit of work executed within a stage.              |
| **Script**           | A list of Linux commands that the GitLab Runner executes.        |
| **GitLab Runner**    | The agent responsible for executing pipeline jobs.               |
| **`.gitlab-ci.yml`** | The configuration file that defines the CI pipeline.             |

---

# Summary

Continuous Integration (CI) in GitLab automates the execution of tasks whenever code is pushed to a repository. The workflow is defined in the **`.gitlab-ci.yml`** file, where you organize work into **stages** and **jobs**. Each job contains a **script** with Linux commands that are executed by a **GitLab Runner**. In this introductory example, the pipeline uses only `echo` commands to simulate the **Build**, **Test**, and **Report** phases, making it easy to understand how GitLab CI pipelines are structured and executed before moving on to real-world tasks such as compiling code, running automated tests, building Docker images, and deploying applications.
